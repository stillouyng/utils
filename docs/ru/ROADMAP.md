## Будущее (после v1.0.0)

- [ ] **`twc-handler` — TTL и отзыв на стороне сервера**: Вспомогательный демон (PAM-модуль или хук `AuthorizedKeysCommand`) на целевом SSH-сервере. Вместо передачи сырых учётных данных, `twc copy --share` выпускал бы ограниченный по времени twc-токен; обработчик на стороне сервера проверяет токен по списку отозванных перед разрешением подключения. Это переносит применение TTL с клиента (рекомендательное) на сервер (жёсткое), делая истечение срока криптографически неизбежным даже если получатель держит расшифрованные учётные данные. Требует демона на каждом целевом сервере — принципиально иная модель развёртывания по сравнению с текущей архитектурой без зависимости от сервера.

## Выпущено

<details><summary><b>SCP</b> — баги </summary>

1. Нет транслирования статуса. Условная проблема, если скачивания/загрузка пройдут успешно, но при ошибке поведение не предсказуемо.
2. Передача директорий не поддерживается. Флаг `-r` не передаётся, поэтому `twc scp` молча падает или выдаёт ошибку если источником является директория.
3. Паника вместо понятной ошибки если бинарник `scp` не найден. Ветка аутентификации по ключу использует `.expect()` при запуске процесса, что крашит программу вместо вывода понятного сообщения.
4. Нет валидации путей перед запуском. Пустые или заведомо неверные пути источника/назначения передаются напрямую в `scp` без предварительной проверки.

</details>

<details><summary><b>SCP — передача файлов без пароля</b></summary>

Обработка scp без ввода пароля: `twc scp <name>`

</details>

<details><summary><b>Обнуление секретов в памяти</b></summary>

Расшифрованные пароли и ключи хранятся как обычные `String`/`Vec<u8>` на куче; Rust не обнуляет память при освобождении. Дамп памяти процесса мог бы их раскрыть. Исправление: оборачивать чувствительные расшифрованные значения в `zeroize::Zeroizing<T>`, чтобы они очищались из памяти как только выходят из области видимости. (Приватный ключ X25519 уже защищён — `x25519-dalek` использует `zeroize` внутри.)

</details>

<details><summary><b>Защита командной строки sshpass</b></summary>

`twc <name>` сейчас запускает `sshpass -p <открытый текст>`, что открывает пароль в командной строке процесса (`/proc/<pid>/cmdline`) другим локальным процессам на время SSH-сессии. Исправление: записывать пароль в пайп и передавать файловый дескриптор читающего конца через `sshpass -d <fd>` — пароль никогда не появляется в командной строке.

</details>

<details><summary><b>Права доступа к конфигурационному файлу</b></summary>

После записи `config.json` ограничить права только для владельца (`0600` на Unix). Зашифрованные блобы в безопасности, но доступный для чтения всеми конфиг раскрывает список профилей, имена пользователей и хосты любому локальному пользователю.

</details>

<details><summary><b>Копирование ключа обмена в буфер обмена</b></summary>

Ключ теперь по умолчанию копируется в буфер обмена.

</details>

<details>
<summary><b>Устранение дублирования запроса мастер-ключа</b></summary>

Когда `twc add` или `twc edit` вызывается с `--password` и `--sudo-password` одновременно, мастер-ключ запрашивается дважды — по одному на каждый секрет — так что пользователь мог случайно зашифровать их разными ключами. Запрашивать один раз и использовать один мастер-ключ для обоих секретов в одном вызове.

</details>

<details>
<summary><b>Исправления TTL</b> — усиление безопасности обмена</summary>

- **Неправильное имя флага `--share`**: примеры в справке и ошибки времени выполнения теперь корректно ссылаются на `--for-key` вместо `--for`.
- **Утечка открытого текста `--share --ttl`**: `twc copy` и `twc copy-sp` теперь блокируют извлечение учётных данных из любого общего профиля.
- **Профили `--share --ttl` не помечаются в хранилище**: `SSHConfig` теперь содержит `shared` и `expires_at`; `twc list` показывает метки `[shared]` / `[shared · expired]`, `twc show` выводит обратный отсчёт до истечения, SSH блокируется после истечения срока, а редактирование заблокировано полностью.
- **Обратная совместимость `--share --ttl`**: TTL-блобы теперь используют префикс `TWC3:`, чтобы старые клиенты (принимающие только `TWC2:`) выдавали жёсткую ошибку вместо тихого импорта с игнорированием `expires_at`. Смотрите [VULNERABILITIES.md](VULNERABILITIES.md) для оставшегося ограничения кооперативного доверия.

</details>

<details>
<summary><b>Обмен профилями — TTL</b></summary>

Расширение `--share` опциональным флагом `--ttl <value>`, например `twc copy <name> --share --for <pubkey> --ttl 30M`. Поддерживаемые единицы: `s` (секунды), `M` (минуты), `H` (часы), `d` (дни), `m` (месяцы), `y` (годы) — обратите внимание на `M` vs `m` во избежание неоднозначности; календарные единицы используют фиксированные множители (`1m = 30d`, `1y = 365d`). Метка истечения встраивается в открытый текст `ShareBlob` перед ECIES-шифрованием, поэтому аутентификация AES-256-GCM делает её защищённой от подделки — никто не может продлить TTL без приватного ключа получателя. `twc add --from-clip` проверяет `now > expires_at` и отклоняет истёкшие блобы до обращения к учётным данным.

</details>

<details>
<summary><b>Обмен профилями</b> — <code>twc copy &lt;name&gt; --share --for &lt;pubkey&gt;</code> / <code>twc add --from-clip</code> / <code>twc share-key</code></summary>

`twc copy <name> --share --for <pubkey>` шифрует переносимый блоб профиля для конкретного получателя с использованием ECIES (эфемерный X25519 + HKDF-SHA256 + AES-256-GCM). Секреты передаются открытым текстом *внутри* зашифрованного конверта — отправитель расшифровывает их локально перед сборкой блоба, поэтому мастер-ключ отправителя никогда не покидает его машину. `twc add --from-clip` расшифровывает приватным ключом получателя и повторно шифрует все секреты под собственным мастер-ключом получателя. Для профилей на основе ключа байты приватного ключа встраиваются в блоб и записываются в `~/.ssh/twc_<name>` при импорте. X25519-пара получателя автоматически генерируется при первом вызове `twc share-key` и хранится зашифрованной (Argon2 + AES-256-GCM) под его мастер-ключом — тот же корень доверия, что и для всего остального.

</details>

<details>
<summary><b>Хранение и копирование sudo-пароля</b> — <code>twc copy-sp &lt;name&gt;</code></summary>

Хранить зашифрованный sudo-пароль для каждого профиля (то же AES-256-GCM + Argon2, что и для SSH-пароля, тот же мастер-ключ). Единственная операция — копирование в буфер обмена. Пример: `twc copy-sp <name>`.

</details>

<details>
<summary><b>Копирование конфигурации</b> — <code>twc copy &lt;name&gt;</code></summary>

Копировать информацию SSH-подключения профиля в буфер обмена. Уведомляет если профиль использует ключ (файл идентификации); иначе запрашивает мастер-ключ и копирует строку подключения с расшифрованным паролем.

</details>

<details>
<summary><b>Уникальность имён конфигов | Имён команд</b> (исправление бага)</summary>

`twc rename` и `twc add` ранее позволяли задать имя профиля совпадающее с зарезервированной подкомандой CLI (например `add`, `list`, `remove`, `rename`, `show`, `edit`, `copy`), что делало профиль недостижимым через `twc <name>`.

</details>
