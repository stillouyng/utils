# Ограничения безопасности

> **Language:** [English](../en/VULNERABILITIES.md) | **Русский**

> Назад к [README](README.md)

Этот документ описывает известные ограничения безопасности twc, которые не могут быть полностью устранены в рамках текущей локальной архитектуры (zero-server, local-only). Они раскрываются прозрачно, чтобы пользователи могли принимать обоснованные решения о доверии.

---

## TTL носит кооперативный, а не криптографический характер

**Затрагивает:** `twc copy --share --ttl`

Получатель общего блоба владеет приватным ключом ECIES, необходимым для его расшифровки. Поскольку расшифровка полностью локальная, целеустремлённый получатель всегда может прочитать содержимое блоба — включая `expires_at` — независимо от того, что применяет прикладной уровень. Конкретно:

- Они могут использовать модифицированный бинарник twc, игнорирующий `expires_at`.
- Они могут скачать старый бинарник twc, выпущенный до появления поддержки TTL.
- Они могут декодировать и изучить блоб вручную с помощью стандартных криптографических библиотек.

**Что twc всё же смягчает:**

| Смягчение                                                   | Что предотвращает                                                  |
|-------------------------------------------------------------|--------------------------------------------------------------------|
| TTL-блобы используют префикс `TWC3:`                        | Старые клиенты падают с ошибкой вместо тихого обхода TTL           |
| `twc copy` / `twc copy-sp` заблокированы для общих профилей | Извлечение учётных данных через команды twc после импорта          |
| SSH заблокирован для истёкших общих профилей                | Удобный доступ после истечения срока для кооперативных получателей |
| `expires_at` аутентифицирован AES-256-GCM                   | Никто не может продлить TTL без приватного ключа получателя        |

**Что не смягчает:**

Получатель, намеренно желающий обойти TTL, может это сделать. TTL в twc — это **соглашение об окне доступа между кооперативными сторонами**, а не жёсткая криптографическая гарантия.

**Реальный фикс:** Применение на стороне сервера через демон `twc-handler` на целевом SSH-сервере (смотрите [ROADMAP.md — Будущее](ROADMAP.md)). Сервер проверял бы ограниченный по времени токен перед разрешением подключения, делая истечение срока применимым даже против получателя, расшифровавшего блоб (В теории).
